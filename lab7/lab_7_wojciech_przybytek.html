Wstęp¶

Celem tego laboratorium jest zapoznanie siz kilkoma metodami wykorzystywanymi do podejmowania decyzji w kontekcie, w ktrym dwa podmioty maj sprzeczne cele, a konkretnie w sytuacji, w ktrej wygrana jednego podmiotu oznacza przegran drugiego podmiotu. Wykorzystamy do tego gr w kko i krzyyk, ktra posiada bardzo proste zasady. Zaprezentowane metody maj jednak znacznie szersze zastosowanie i mog by wykorzystywane w szerokim spektrum problemw decyzyjnych.

Zastosowane metody to:

  • losowe przeszukiwanie przestrzeni decyzji,
  • heurystyczne przeszukiwanie przestrzeni decyzji,
  • algorytm minimax,
  • algorytm alpha-beta,
  • przeszukiwanie w oparciu o metod Monte Carlo.

Na kocu laboratorium zrobimy turniej graczy zaimplementowanych z uyciem tych metod i zobaczymy, ktra metoda razi sobie najlepiej w grze w kko i krzyyk.

PLAYER_1 = -1
PLAYER_2 = 1


class TicTacToe:
    def __init__(self, player1, player2):
        self.board = [
            [0, 0, 0],
            [0, 0, 0],
            [0, 0, 0],
        ]
        self.player1 = player1
        self.player2 = player2
        self.chars = {0: " ", PLAYER_1: "O", PLAYER_2: "X"}

    def play(self, verbose=True):
        self.print_board(verbose)
        for i in range(4):
            self.player1.move(self.board, PLAYER_1)
            self.print_board(verbose)
            if check_for_end(self.board, PLAYER_1):
                if verbose:
                    print("player 1 wins")
                return "win"
            self.player2.move(self.board, PLAYER_2)
            self.print_board(verbose)
            if check_for_end(self.board, PLAYER_2):
                if verbose:
                    print("player 2 wins")
                return "loss"
        self.player1.move(self.board, PLAYER_1)
        self.print_board(verbose)
        if check_for_end(self.board, PLAYER_1):
            if verbose:
                print("player 1 wins")
            return "win"
        else:
            if verbose:
                print("draw")
            return "draw"

    def print_board(self, verbose):
        if not verbose:
            return
        str_line = "---------------"
        print("\n" + str_line)
        for row in self.board:
            for cell in row:
                symbol = self.chars[cell]
                print(f"| {symbol} |", end="")
            print("\n" + str_line)


def check_for_end(board, player):
    return (
        check_rows(board, player)
        or check_cols(board, player)
        or check_diagonals(board, player)
    )


def check_rows(board, player):
    for i in range(3):
        if board[i][0] == board[i][1] == board[i][2] == player:
            return True
    return False


def check_cols(board, player):
    for i in range(3):
        if board[0][i] == board[1][i] == board[2][i] == player:
            return True
    return False


def check_diagonals(board, player):
    return (board[0][0] == board[1][1] == board[2][2] == player) or (
        board[0][2] == board[1][1] == board[2][0] == player
    )

from abc import ABC, abstractmethod


class PlayerIntreface(ABC):
    @abstractmethod
    def move(self, board, player):
        pass

    @property
    def name(self):
        return type(self).__name__

Podejmowanie decyzji oparte o heurystykę¶

W kadym nowym problemie warto sprawdzi proste podejcie heurystyczne, wykorzystujce elementarn wiedz o problemie. Sama gra w kko i krzyyk, jak wiadomo, nie jest zbyt skomplikowana. Jest w niej najwyej 9 moliwych ruchw, mniej ni $3^9 - 1$ moliwych kocowych stanw planszy, a naiwnie liczc, gr mona rozegra na $9!$ sposobw. Dziki tam mocnemu uproszczeniu istnieje tu strategia optymalna, gwarantujca w najgorszym wypadku remis:

Dla tej gry istnieje optymalna startegia tzn. w najgorszym przypadku zremisujemy.

  1. Win: If the player has two in a row, they can place a third to get three in a row.
  2. Block: If the opponent has two in a row, the player must play the third themselves to block the opponent.
  3. Fork: Cause a scenario where the player has two ways to win (two non-blocked lines of 2).
  4. Blocking an opponent's fork: If there is only one possible fork for the opponent, the player should block it. Otherwise, the player should block all forks in any way that simultaneously allows them to make two in a row. Otherwise, the player should make a two in a row to force the opponent into defending, as long as it does not result in them producing a fork. For example, if "X" has two opposite corners and "O" has the center, "O" must not play a corner move to win. (Playing a corner move in this scenario produces a fork for "X" to win.)
  5. Center: A player marks the center. (If it is the first move of the game, playing a corner move gives the second player more opportunities to make a mistake and may therefore be the better choice; however, it makes no difference between perfect players.)
  6. Opposite corner: If the opponent is in the corner, the player plays the opposite corner.
  7. Empty corner: The player plays in a corner square.
  8. Empty side: The player plays in a middle square on any of the four sides.

Implementacja takiego bota byaby jednak niezbyt ciekawa, a na dodatek specyficzna tylko dla tego konkretnego problemu. Dlatego my zajmiemy si eksploracj drzewa gry, a wic moliwych stanw oraz przej midzy nimi. Na dobry pocztek ulepszymy naszego losowego bota trywialn heurystyk - jeeli to moliwe, ma wybra ruch wygrywajcy, a jeeli nie, to losowy.

Mona by te zada pytanie, czemu nie wykorzystamy tutaj ciekawszego problemu, jak np. szachy. Odpowied jest prosta - nie mamy na to czasu, a konkretnie wiecznoci. Shannon obliczy doln granic zoonoci drzewa gry na $10^{120}$.

        if row != -1 and col != -1:
            board[row][col] = player
            return
        while True:
            row = random.randint(0, 2)
            col = random.randint(0, 2)

                board[row][col] = player

    def _check_win_all(self, board, player):
        row, col = self._check_win_in_rows(board, player)
        if row != -1 and col != -1:
            return row, col
        row, col = self._check_win_in_columns(board, player)
        if row != -1 and col != -1:
            return row, col

        if row != -1 and col != -1:

        return -1, -1

Algorytm minimax¶

Zaiplemetujemy teraz algorytm minimax. Minimalizuje on nasze maksymalne straty lub maksymalizuje minimalne zyski (maksmin). Poprzednie 2 kroki nas do tego zbliay. Algorytm wywodzi si z teorii gier o sumie zerowej (gdzie wygrana ma warto 1, przegrana -1, a remis 0 - w takich grach wygrana jednego gracza, oznacza, e drugi gracz przegra, czyli nie ma strategii, ktra prowadziaby do sytuacji win-win).

Twierdzenie o minimaksie (minimax theorem) mwi, e dla kadej gry dwuosobowej gry o sumie zerowej istnieje warto V i mieszana strategia dla kadego gracza, takie, e:

  • biorc pod uwag strategi gracza drugiego, najlepsz moliw spat dla gracza pierwszego jest V,
  • biorc pod uwag strategi gracza pierwszego, najlepsz moliw spat dla gracza drugiego jest -V.

Kady gracz minimalizuje maksymaln moliw spat dla swojego przeciwnika poniewa gra jest gr o sumie zerowej, wynikiem tego algorytmu jest rwnie maksymalizowanie swojej minimalnej spaty.

Powysza ilustracja przedstawia fragment analizy kocowej partii gry w kko i krzyyk. Max oznacza tur, w ktrej gracz wybiera ten spord dostpnych ruchw, ktry da maksymaln spat, natomiast min oznacza tur, w ktrej gracz wybiera ruch, ktry da minimaln spat. Inaczej - max to ruch, z punktu wiedzenia gracza, dla ktrego chcemy, eby wygra, a min ruch gracza, ktry chcemy eby przegra.

Minimax jest algorytem rekurencyjnym, w ktrym licie drzewa moliwych ruchw oznaczaj zakoczenie gry, z przypisan do nich wartoci z punktu wiedzenia gracza, ktrego ruch jest w korzeniu drzewa moliwych ruchw. We wczeniejszych wzach - w zalenoci czy jest tura gracza max, czy min, bd wybierane te gazie, ktre maksymalizuj, bd minimalizuj warto spaty.

Przykadowo - w pierwszym wierszu mamy trzy strzaki i gracza max, dlatego gracz ten wybierze pierwszy moliwy ruch, bo daje on maksymaln spat. W drugim wierwszu w rodkowej kolumnie mamy dwie strzaki. Gracz min wybierze zatem strzak (ruch) prawy, poniewa minimalizuje on spat (0). Analogicznie w drugim wierszu po prawej stronie wybierana jest prawa strzaka, poniewa daje ona warto mimaln (-1).

Poniszy diagram zawiera to samo drzewo analizy ruchw, gdzie pozostawiono wycznie wartoci spaty dla poszczeglnych wzw.

Algorytm minimax jest nastpujcy:

  1. zainicalizuj warto na $-\infty$ dla gracza ktrego spata jest maksymalizowana i na $+\infty$ dla gracza ktrego spata jest minimalizowana,
  2. sprawd czy gra si nie skoczya, jeeli tak to ewaluuj stan gry z punktu widzenia gracza maksymalizujcego i zwr wynik,
  3. dla kadego moliwego ruchu kadego z graczy wywoaj rekurencyjnie minimax:
    1. przy maksymalizacji wyniku, zwiksz wynik, jeli otrzymany wynik jest wikszy od dotychczas najwikszego wyniku,
    2. przy minimalizacji wyniku, pomniejsz wynik, jeli otrzymany wynik jest mniejszy od dotychczas najmniejszego wyniku,
    3. zwr najlepszy wynik.

Algorytm ten wymaga dodatkowo funkcji ewaluujcej, ktra oceni stan gry na kocu. Uznajemy, e zwycistwo to +1, przegrana -1, a remis 0.

from math import inf


class MinimaxPlayer(Blocking):
    def move(self, board, player):
        row, col = self.get_best_move(player, board)
        board[row][col] = player

    def get_best_move(self, player, board):
        if is_board_empty(board):
            return 1, 1

        best_eval = -inf
        best_move = (-1, -1)
        for row in range(3):
            for col in range(3):
                if can_make_move(board, row, col):
                    board[row][col] = player
                    curr_eval = self.minimax(board, -player, False)
                    if curr_eval > best_eval:
                        best_eval = curr_eval
                        best_move = (row, col)
                    board[row][col] = 0
        return best_move

    def minimax(self, board, player, is_maximizing_player):
        curr_eval = self.evaluate(board, player if is_maximizing_player else -player)
        if is_board_full(board) or curr_eval != 0:
            return curr_eval

        if is_maximizing_player:
            best_eval = -inf
            for row in range(3):
                for col in range(3):
                    if can_make_move(board, row, col):
                        board[row][col] = player
                        curr_eval = self.minimax(board, -player, False)
                        best_eval = max(best_eval, curr_eval)
                        board[row][col] = 0
            return best_eval
        else:
            best_eval = inf
            for row in range(3):
                for col in range(3):
                    if can_make_move(board, row, col):
                        board[row][col] = player
                        curr_eval = self.minimax(board, -player, True)
                        best_eval = min(best_eval, curr_eval)
                        board[row][col] = 0
            return best_eval

    def evaluate(self, board, player):
        if check_for_end(board, player):
            return 1
        elif check_for_end(board, -player):
            return -1
        return 0


def is_board_full(board):
    return sum(list(map(lambda x: abs(x[0]) + abs(x[1]) + abs(x[2]), board))) == 9


def is_board_empty(board):
    return sum(list(map(lambda x: abs(x[0]) + abs(x[1]) + abs(x[2]), board))) == 0

Alpha-beta pruning¶

Widzimy, e nasz poprzedni gracz jest waciwie idealny, bo w kocu sprawdza cae drzewo gry. Ma tylko w zwizku z tym wad - dla wikszoci problemw wykonuje si wieczno. Dlatego te zastosujemy wane ulepszenie algorytmu minimax, nazywane alfa-beta pruning.

W przypadku klasycznego minimaxu ewaluujemy kad moliw ciek gry. Alfa-beta pruning, jak i wiele innych metod, opiera si na "przycinaniu" drzewa, czyli nie ewaluujemy tych odng drzewa, co do ktrych wiemy, e nie da ona lepszego wyniku ni najlepszy obecny. W niektrych wypadkach, np. w dobrej implementacji dla szachw, potrafi zredukowa liczb rozwaanych cieek nawet o 99.8%.

Do poprzedniego algorytmu dodajemy 2 zmienne, $\alpha$ i $\beta$:

  • $\alpha$ przechowuje najlepsz warto dla gracza maksymalizujcego swj wynik,
  • $\beta$ przechowuje najlepsz warto dla gracza minimalizujcego swj wynik.

Dziki tej informacji moemy przerwa sprawdzanie danej gazi, kiedy $\alpha$ jest wiksza od $\beta$. Oznacza to bowiem sytuacj, w ktrej najlepszy wynik gracza maksymalizujcego jest wikszy ni najlepszy wynik gracza minimalizujcego.

class AlphaBetaPlayer(MinimaxPlayer):
    def minimax(self, board, player, is_maximizing_player, alpha=-inf, beta=inf):
        curr_eval = self.evaluate(board, player if is_maximizing_player else -player)
        if is_board_full(board) or curr_eval != 0:
            return curr_eval

        if is_maximizing_player:
            best_eval = -inf
            for row in range(3):
                for col in range(3):
                    if can_make_move(board, row, col):
                        board[row][col] = player
                        curr_eval = self.minimax(board, -player, False, alpha, beta)
                        best_eval = max(best_eval, curr_eval)
                        board[row][col] = 0
                        alpha = max(alpha, best_eval)
                        if alpha >= beta:
                            return best_eval
            return best_eval
        else:
            best_eval = inf
            for row in range(3):
                for col in range(3):
                    if can_make_move(board, row, col):
                        board[row][col] = player
                        curr_eval = self.minimax(board, -player, True, alpha, beta)
                        best_eval = min(best_eval, curr_eval)
                        board[row][col] = 0
                        beta = min(beta, best_eval)
                        if alpha >= beta:
                            return best_eval
            return best_eval

Monte Carlo Tree Search (MCTS)¶

Metody Monte Carlo polegaj na wprowadzeniu losowoci i przyblianiu za jej pomoc rozwiza dla trudnych problemw. Dla gry w kko i krzyyk nie jest to co prawda niezbdne, ale dla bardziej skomplikowanych gier, jak szachy czy go, ju zdecydowanie tak.

Oglna metoda MCTS skada si z 4 etapw:

  • selekcji - wybieramy najlepsze dziecko, a dotrzemy do licia,
  • ekspansji - jeeli nie moemy dokona selekcji, rozwijamy drzewo we wszystkich moliwych kierunkach z wza,
  • symulacji - po ekspansji wybieramy wze do przeprowadzenia symulacji gry a do koca,
  • wstecznej propagacji - kiedy dotrzemy do koca, ewaluujemy wynik gry i propagujemy go w gr drzewa.

W naszym wypadku wystarczy nieco prostszy algorytm Pure Monte Carlo Tree Search (Pure MCTS), w ktrym realizujemy tylko symulacj i wsteczn propagacj. Dla kadego moliwego ruchu w danej rundzie przeprowadzamy N symulacji oraz obliczamy prawdopodobiestwo zwycistwa/remisu/przegranej dla kadego z moliwych ruchw, a nastpnie wybieramy najlepszy ruch.

ITERATIONS = 1000


class MonteCarloPlayer(MinimaxPlayer):
    def move(self, board, player):
        row, col = self.get_best_move(player, board)
        board[row][col] = player

    def get_best_move(self, player, board):
        if is_board_empty(board):
            return 1, 1

        best_eval = [-inf, -inf, inf]
        best_move = (-1, -1)
        for row in range(3):
            for col in range(3):
                if can_make_move(board, row, col):
                    board[row][col] = player
                    curr_eval = self.mcts_evaluate_move(board, player)
                    if self.compare_eval(curr_eval, best_eval):
                        best_eval = curr_eval
                        best_move = (row, col)
                    board[row][col] = 0
        return best_move

    def mcts_evaluate_move(self, board, player):
        possible_moves = self.get_possible_moves(board)
        scores = [0, 0, 0]

        for _ in range(ITERATIONS):
            possible_moves_copy = [x for x in possible_moves]
            board_copy = [[cell for cell in row] for row in board]
            current_player = -player

            while self.evaluate(board_copy, player) == 0 and not is_board_full(
                board_copy
            ):
                move = random.randint(0, len(possible_moves_copy) - 1)
                row, col = possible_moves_copy.pop(move)
                board_copy[row][col] = current_player
                current_player = -current_player

            result = self.evaluate(board_copy, player)
            if result == 1:
                scores[0] += 1
            elif result == 0:
                scores[1] += 1
            else:
                scores[2] += 1

        return scores

    def get_possible_moves(self, board):
        possible_moves = []
        for row in range(3):
            for col in range(3):
                if can_make_move(board, row, col):
                    possible_moves.append((row, col))
        return possible_moves

    def compare_eval(self, eval1, eval2):
        if eval1[0] == eval2[0]:
            return eval1[1] > eval2[1]
        else:
            return eval1[0] > eval2[0]

Pytania kontrolne (1 punkt)¶

  1. Jaki algorytm z omawianych mgby si sprawdzi w grze go i dlaczego?
  2. Czy minimax jest optymalny dla gry w szachy i dlaczego?
  3. Jakie s moliwe zastosowania podanych algorytmw poza grami?
  4. Jaka jest przewaga obliczeniowa MCTS nad pozostaymi metodami?
  • Sprawdzanie wszystkich moliwoci w tak skomplikowanej grze, jak go (ok. $10^{172}$ moliwych partii), byoby bardzo kosztowne dla minimaxu, a nawet alpha-beta pruning mgby wykonywa si wzgldnie dugo, dlatego najlepszym rozwizaniem byoby uycie Monte Carlo Tree Search.
  • Zdecydowanie nie, np. w pocztkowej fazie gry sprawdzamy wszystkie moliwoci, a bardziej zaawansowany algorytm mgby wykorzysta wiedz dotyczc teorii debiutw, aby znacznie zoptymalizowa ten proces.
  • np. MCTS do Scheduling Problem
  • Wybranie odpowiednio duej prbki i na jej podstawie podanie odpowiedzi w algorytmie MCTS da rwnie dobre wyniki co dokadne sprawdzenie wszystkich moliwoci (nawet tylko tych, ktre maj sens jak w alpha-beta pruningu), a wic mniejszym kosztem obliczeniowym otrzymujemy rwnie dobre wyniki, co jest oczywist zalet, jednym problemem jest wybranie odpowiednio duej prbki.

Zadanie dodatkowe¶

Rozwi kod algorytmu MCTS tak, aby dziaa z oglnymi zasadami, a nie w formie uproszczonego Pure MCTS.

Zastosuj prosty, ale bardzo skuteczny sposb selekcji UCT (Upper Confidence Bound 1 applied to trees), bdcy wariantem bardzo skutecznych metod UCB, stosowanych m.in. w podejmowaniu decyzji, uczeniu ze wzmocnieniem i systemach rekomendacyjnych. Polega na wyborze tego wza, dla ktrego nastpujce wyraenie ma maksymaln warto:

$$ \large \frac{w_i}{n_i} + c\sqrt\frac{\ln N_i}{n_i} $$

gdzie:

  • $w_i$ to liczba zwycistw dla wza po rozwaeniu $i$-tego ruchu
  • $n_i$ to czna liczba symulacji przeprowadzonych dla wza po $i$-tym ruchu,
  • $N_i$ oznacza cakowit liczb symulacji przeprowadzon po $i$-tym ruchu dla wza-rodzica aktualnie rozwaanego wza,
  • $c$ to hiperparametr, wedle teorii powinien mie warto $\sqrt{2}$.

O uzasadnieniu tego wzoru moesz wicej przeczyta tutaj.

Jeeli chcesz, moesz zastosowa ten algorytm dla bardziej skomplikowanej gry, jak np. warcaby czy szachy.
